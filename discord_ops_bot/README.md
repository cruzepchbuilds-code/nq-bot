# Ops Discord Bot — Multi-Agent Dev/Ops Server

A Discord server setup for the `nq_bot_final` repo (and any other project),
with **5 separate bot accounts** — each with its own name, avatar, server
role, and channel:

| Agent | Bot identity | Role | Channel | What it does |
| --- | --- | --- | --- | --- |
| **Ops Agent** | "Ops Agent" 🔘 gear avatar | `Ops Agent` | `#ops-log`, `#bot-status` | `/ping`, `/status`, `/agents`, heartbeat every 30 min |
| **CI Agent** | "CI Agent" ✅ check avatar | `CI Agent` | `#ci-cd` | Posts GitHub Actions build/test results; `/ci status`, `/ci runs` |
| **Review Agent** | "Review Agent" 🔍 avatar | `Review Agent` | `#code-review` | Posts new PRs needing review; `/review` lists open PRs |
| **Deploy Agent** | "Deploy Agent" 🚀 avatar | `Deploy Agent` | `#deployments` | Tracks deploy history; `/deploy status`, `/deploy log` |
| **Research Agent** | "Research Agent" 📊 avatar | `Research Agent` | `#research-alerts` | Watches `brain/*.md` / `results/*.md` for new backtest/walk-forward reports; `/research`, `/run <job>` |

It also creates `#general` and `#announcements` in a "💬 GENERAL" category,
and `#agent-general` — a shared channel under "🤖 OPS AGENTS" where all 5
agents (and humans) can post.

Avatars are generated by `generate_avatars.py` -> `assets/avatars/*.png`
(gear/check/magnifier/rocket/bar-chart icons, color-matched per agent).

a **read-only feed**: `@everyone` can view/read history but not post; only
the matching "desk" role (held by the relevant agent bot) can post.

| Category | Channel | Desk role | Owner | Purpose |

## Multi-bot setup (5 separate Discord Applications)

The **Ops Agent** is already live (renamed from the original bot, gear
avatar set, `Ops Agent` role assigned to itself). To get the other 4 agents
online as their own named/avatared bots:

### For each of `ci`, `review`, `deploy`, `research`:

1. **Create a new Application**: https://discord.com/developers/applications
   -> **New Application** -> name it anything (we'll rename it via API).
2. **Bot** tab -> **Reset Token** -> copy it.
3. Paste it into `.env` as `DISCORD_BOT_TOKEN_CI` (or `_REVIEW`, `_DEPLOY`,
   `_RESEARCH`).
4. Set its name + avatar automatically:
   ```bash
   ./venv/bin/python set_identity.py ci
   ```
5. Get its invite link and open it (pick the **CruzCapital** server):
   ```bash
   ./venv/bin/python invite_link.py ci
   ```
6. Once it's joined the server, grant it its matching role:
   ```bash
   ./venv/bin/python assign_role.py ci
   ```
   (If this returns a 403 about role hierarchy, drag the **Ops Agent** role
   above the other agent roles in Server Settings -> Roles, then retry.)
7. Start it:
   ```bash
   ./venv/bin/python bot.py --agent ci
   ```

Repeat for `review`, `deploy`, `research`. Or, once all 4 tokens are in
`.env`, bring everything up at once with:

```bash
./run_all.sh
```

(logs go to `logs/<agent>.log`, Ctrl-C stops all 5)

## Persistence (launchd) — keep all 5 bots running

`run_all.sh`/`nohup` only survives until the terminal/session ends. For a
true always-on setup (auto-start on login, auto-restart on crash), install
each agent as a macOS LaunchAgent:

```bash
./venv/bin/python setup_launchd.py            # install + (re)start all 5
./venv/bin/python setup_launchd.py --status   # show launchctl status
./venv/bin/python setup_launchd.py --uninstall  # stop + remove all 5
```

This writes one plist per agent to
`~/Library/LaunchAgents/com.cruzcapital.discordbot.<agent>.plist` with
`RunAtLoad` + `KeepAlive` (15s throttle). Logs go to
`~/Library/Logs/CruzCapitalBots/<agent>.log` / `.err.log` — **not**
`discord_ops_bot/logs/`, because launchd's `posix_spawn` fails with exit 78
(`EX_CONFIG`) when stdout/stderr redirect into `~/Desktop/...` (macOS TCC
blocks launchd's fd setup for the protected Desktop folder, even though the
running process can read/write there fine once spawned). The
`ProgramArguments` therefore run `/bin/bash -c "exec ./venv/bin/python
bot.py --agent <name>"` with `WorkingDirectory` set to this directory.

Idempotent — re-run any time after editing `bot.py`/`.env`/cogs to reload
and restart all 5 agents.

## Permissions & anti-spam audit (June 2026)

A permissions pass on the CruzCapital server found two open channels under
**🤖 OPS AGENTS** where regular members had full posting rights:

- `#agent-general` — `@everyone` had the same `FULL_ALLOW` as the 5 agent
  roles (view+send+read-history)
- `#bot-status` — had **no** permission overwrites at all, so it inherited
  `@everyone`'s guild-level `SEND_MESSAGES`

Both are now locked to **read-only for `@everyone`** (view + read-history,
`SEND_MESSAGES` denied) — same pattern as the other 14 agent/quant-desk
channels. Only the matching agent bot(s) can post.

Also added two AutoMod rules (on top of the 5 already on the server —
"Block Suspected Spam Content", word filters, etc.):

- **Anti-Spam: Mention Spam Protection** — blocks messages with 5+ mentions,
  raid-protection enabled
- **Anti-Spam: Block Discord Invite Links** — blocks `discord.gg/...` and
  `discord.com/invite/...` links

Both exempt the 5 agent bot roles + CEO/Owner/Co-Owner so normal bot/staff
posts aren't affected.

**Stale-role audit**: the server has two roles named "Agents" (one with
`Administrator`, one with `Administrator + Manage Roles`) plus
`Agent2`/`Agent 3`/`Agent 4`/`Agent 5` (broad send/webhook perms). These
looked like leftover admin-bypass roles that could defeat all the channel
locks above. Audited via `/guilds/{id}/members/search` across all 28
members on the server: **these roles are held only by our 5 agent bots**
(Discord auto-creates a "managed" integration role per bot app — granted
when each app was invited with `Administrator` — plus a shared "Agents"
role all 5 bots hold). No human member holds any of them. No cleanup
needed.

## Verification (RestoreCord)

`setup_verification.py` scaffolds a verification gate for **new members**:

- Creates a **"Verified"** role (`ROLE_VERIFIED` in `.env`)
- Creates a read-only **`#verify`** channel (`CHANNEL_VERIFY` in `.env`)
  under 💬 GENERAL, with an embed explaining the verification step

```bash
./venv/bin/python setup_verification.py
```

Idempotent — safe to re-run. Once `RESTORECORD_VERIFY_URL` is set in `.env`
(see below), re-running it adds a **"Verify Now"** link button to the
`#verify` embed.

### RestoreCord setup (manual — third-party site)

This part happens on restorecord.com + the Discord Developer Portal, not
via this repo's bots:

1. Sign up / log in at **restorecord.com** and create a new server entry
   for CruzCapital (Guild ID: see `DISCORD_GUILD_ID` in `.env`).
2. In the **Discord Developer Portal**, pick an Application to use for OAuth
   (can be a dedicated new app — don't reuse the 5 agent bot apps, to keep
   their tokens isolated). Under **OAuth2**, copy the **Client ID** and
   **Client Secret**.
3. Add RestoreCord's callback URL as a **Redirect URI** on that app (exact
   URL is shown in the RestoreCord dashboard for your server).
4. In the RestoreCord dashboard, paste the Client ID/Secret, and set the
   **"Verified Role"** to the `ROLE_VERIFIED` ID printed by
   `setup_verification.py` (also stored in `.env`).
5. Invite **RestoreCord's bot** to the server using the invite link from
   its dashboard. In **Server Settings -> Roles**, drag RestoreCord's bot
   role **above** "Verified" — a bot can only grant roles below its own
   highest role.
6. Copy the verification page URL RestoreCord gives you, set
   `RESTORECORD_VERIFY_URL=...` in `.env`, then re-run
   `./venv/bin/python setup_verification.py` to add the "Verify Now" button.

### Gating channels behind "Verified" (do this AFTER step 6 is tested)

Scope is **new members only** — existing members/customers must keep their
current access, so don't blanket-deny `@everyone` across the server. Steps:

1. **Grandfather current members**: bulk-assign the "Verified" role to every
   existing member (Server Settings -> Members -> select all -> Add Role,
   or a script using `/guilds/{id}/members/search` + `PUT
   /guilds/{id}/members/{user}/roles/{role}` for each).
2. For each channel that should be hidden from un-verified new joiners, add
   a permission overwrite: `@everyone` deny `VIEW_CHANNEL`, "Verified" allow
   `VIEW_CHANNEL` (+ whatever else `@everyone` currently has there).
3. Leave `#verify` (and any welcome/rules channel) visible to `@everyone` so
   new members can find the verification link.

New members land with only `@everyone` until RestoreCord grants them
"Verified", so they'll see `#verify` (+ anything else still open to
`@everyone`) until they complete verification — existing members are
unaffected because they already hold "Verified" from step 1.

## 1. Create the Discord Application + Bot

1. Go to https://discord.com/developers/applications -> **New Application**.
2. **Bot** tab -> **Reset Token** -> copy it (this is `DISCORD_BOT_TOKEN`).
3. Under **Privileged Gateway Intents**, you can leave everything off — this
   bot only uses slash commands (no message content needed).
4. **OAuth2 -> URL Generator**:
   - Scopes: `bot`, `applications.commands`
   - Bot Permissions: `Administrator` (simplest for first-run setup — it
     needs to create channels/roles/webhooks). You can narrow this down to
     `Manage Channels`, `Manage Roles`, `Manage Webhooks`, `Send Messages`,
     `Embed Links`, `Use Slash Commands` afterwards.
5. Open the generated URL, pick your server, authorize.

## 2. Get your Server (Guild) ID

Discord app -> User Settings -> Advanced -> enable **Developer Mode**.
Right-click your server icon -> **Copy Server ID** -> this is `DISCORD_GUILD_ID`.

## 3. Configure

```bash
cd discord_ops_bot
cp .env.example .env
# edit .env: set DISCORD_BOT_TOKEN and DISCORD_GUILD_ID at minimum
```

Optional but recommended:
- `GITHUB_TOKEN` — a fine-grained PAT with read access to your repo, for
  `/ci` and `/review` commands and the polling loops.
- `GITHUB_REPO` — defaults to `cruzepchbuilds-code/nq_bot_final`.
- `GITHUB_WEBHOOK_SECRET` — any random string; set the same value in the
  GitHub repo's webhook config.

## 4. Install deps

A venv is already set up at `discord_ops_bot/venv/`. If you need to
recreate it:

```bash
python3 -m venv venv
./venv/bin/pip install -r requirements.txt
```

## 5. Provision the server (channels, roles, webhooks) — run once

```bash
./venv/bin/python setup_server.py
```

This creates the categories/channels/roles listed above, creates incoming
webhooks for `#ci-cd`, `#code-review`, `#deployments`, `#research-alerts`,
and writes the resulting channel IDs + webhook URLs back into `.env`.
Safe to re-run — it skips anything that already exists.

## 6. Run the agents (slash commands + heartbeat + watchers)

Each agent is its own bot process/identity (see "Multi-bot setup" above):

```bash
./venv/bin/python bot.py --agent ops        # /ping /status /agents + heartbeat
./venv/bin/python bot.py --agent ci         # /ci status|runs + 5-min Actions poll
./venv/bin/python bot.py --agent review     # /review + 10-min open-PR poll
./venv/bin/python bot.py --agent deploy     # /deploy status|log
./venv/bin/python bot.py --agent research   # /research /run + 5-min brain/*.md watch
```

Or `./run_all.sh` to launch every agent that has a token configured.

Keep these running (e.g. in `tmux`, `screen`, or as `launchd`/systemd
services) for the polling/heartbeat loops and slash commands to work.

## 7. (Optional) Run the webhook relay

For instant GitHub push/PR/CI notifications without polling, and for
piping deploy/research events from scripts that aren't the bot process:

```bash
./venv/bin/uvicorn webhook_server:app --host 0.0.0.0 --port 8787
```

Then in GitHub: **Settings -> Webhooks -> Add webhook**
- Payload URL: `http://<your-host>:8787/github`
- Content type: `application/json`
- Secret: same as `GITHUB_WEBHOOK_SECRET` in `.env`
- Events: Pushes, Pull requests, Workflow runs

### Reporting a deploy from a script

```bash
./venv/bin/python report_deploy.py --target prod --version v7 \
    --status success --message "Deployed v7 config (PF 2.14 OOS)"
```

### Posting a research update from a script

```bash
curl -X POST http://localhost:8787/research \
  -H "Content-Type: application/json" \
  -d '{"title": "Asia S1 research done", "body": "OOS PF 3.54, n=35 — see brain/asia_research.md"}'
```

(The Research Agent's 5-minute file watcher in `bot.py` already does this
automatically for any `brain/*.md` / `results/*.md` file that changes —
this endpoint is for ad-hoc pushes from other tooling.)

## Extending with new agents

Add a new file to `cogs/`, e.g. `cogs/my_agent.py`, with an
`async def setup(bot)` that calls `bot.add_cog(...)`. Add an entry to
`AGENTS` in `shared/config.py` (display name, cog path, role name), add a
matching icon to `generate_avatars.py`, and an entry in `OPS_CHANNELS` in
`setup_server.py` plus `CHANNEL_ENV_KEYS`/`WEBHOOK_ENV_KEYS` if it needs its
own channel. Then: `setup_server.py` -> create the Discord Application ->
`set_identity.py` / `invite_link.py` / `assign_role.py` -> `bot.py --agent <name>`.
